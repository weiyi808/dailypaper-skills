> **执行环境约束（重要）**：
> - 本 skill 运行在 Cursor agent 内，必须直接使用 Cursor 工具链执行。

> - 所谓“调用 `/paper-reader` skill”指：在当前会话内读取并遵循 `paper-reader/SKILL.md` 的规则逐篇处理，而不是启动外部 CLI。

# 读文件夹论文（Batch Folder Papers）

你是用户的“目录批量读论文入口”。目标是：用户只给一个目录路径，你负责递归读取其中论文文件，逐篇生成笔记并更新概念库；**并在每批/全量结束后合并三份 SLAM 总览**（见 Step 4.1），不需要用户手动逐篇下指令。

## Step 0: 读取共享配置

先读取 `../_shared/user-config.json`，如果 `../_shared/user-config.local.json` 存在，再用它覆盖默认值。

显式生成并在后续统一使用这些变量：

- `VAULT_PATH`
- `NOTES_PATH`
- `CONCEPTS_PATH`
- `AUTO_REFRESH_INDEXES`
- `GIT_COMMIT_ENABLED`
- `GIT_PUSH_ENABLED`

其中：

- `NOTES_PATH = {VAULT_PATH}/{paper_notes_folder}`
- `CONCEPTS_PATH = {NOTES_PATH}/{concepts_folder}`
- `GIT_PUSH_ENABLED` 只有在 `GIT_COMMIT_ENABLED=true` 时才可能为真

## Step 1: 解析输入与目录检查

1. 从用户输入中提取目录路径（绝对路径优先）。
2. 若是相对路径，基于当前工作区解析为绝对路径。
3. 检查目录是否存在且可读；不存在则报错并停止。
4. 若用户同时给了模式词（如“快速看一下/批判性分析”），记录为本次批处理模式；默认使用完整解析模式。
5. 若用户未指定批量规模，默认全量；可支持：
   - `limit`: 仅处理前 N 篇（按排序后顺序）
   - `resume`: 跳过已完成项继续
   - `include/exclude`: 按文件名关键词过滤

## Step 2: 递归扫描论文文件

递归扫描目标目录，收集候选论文文件，按文件名排序后处理：

- 支持：`.pdf`（主格式）
- 可选支持：`.html`、`.md`（当内容明显是论文正文时）
- 明确跳过：图片、压缩包、代码文件、日志文件等非论文文件

扫描实现要求（避免漏扫）：

1. 优先使用 Python `os.walk`/`pathlib.rglob` 扫描，不依赖单一 IDE 索引结果。
2. 必须正确处理包含空格、中文、括号、冒号、Unicode、换行符的文件名。
3. 必须去重（真实路径归一化后去重），并保留稳定排序。
4. 若工具扫描结果异常偏少，应自动回退到 Python 扫描并继续，不得静默只处理 1 篇。

输出扫描统计：

- 候选论文总数
- 可处理文件数
- 跳过文件数（附原因）

若可处理文件数为 0，告知用户目录内未发现可读论文并停止。

## Step 3: 逐篇调用 paper-reader（必须）

对每个可处理论文文件，必须调用 `/paper-reader` skill 逐篇处理，禁止手写简化版笔记。

执行规则：

1. 每篇论文都传入实际文件路径（优先绝对路径）。
2. 使用用户指定模式（若有），否则默认完整解析。
3. paper-reader 负责：
   - 结构化论文笔记生成
   - 概念库缺失项补全
   - **文献树 / 挑战洞察 / SLAM 知识库联动**（见 `paper-reader` **§5.5**；批量场景可由本 skill Step 4 统一合并）
4. 对失败项记录错误，不影响后续文件继续处理（容错继续）。
5. 每篇完成后立即记录进度（内存或临时文件），支持中断后恢复。

> **易漏点（本 skill 必须兜底）**：逐篇子任务常**只写 `_待整理` 笔记**而**未改** `{NOTES_PATH}` 下三份总览（`SLAM-文献树.md`、`SLAM-挑战洞察树.md`、`SLAM研究知识库.md`）。**不得**假设 paper-reader 已写入总览；**必须在 Step 4 统一合并**（见下）。

在 Cursor 中的实际执行方式：

- 对每个文件，向当前会话发起等价单篇请求（例如“读一下这篇论文 {绝对路径}”），并遵循 `paper-reader` 规范直接落地。
- **不得**调用任何外部 `claude` 命令来“转发”处理。
- 若单篇处理失败，记录错误并继续下一篇，最后统一汇总。

> 铁律：目录里识别出的可处理论文，必须全部尝试处理，不能因为耗时而静默跳过。

## Step 4: 批处理完成后的统一收尾

### 4.1 SLAM 总览三文件（强制，与 paper-reader §5.5 对齐）

在刷新 MOC **之前**，根据**本批成功生成**的笔记（路径多在 `{NOTES_PATH}/_待整理/`，或已迁入子目录），**更新或合并**以下文件（均位于 `{NOTES_PATH}/` 根下）：

| 文件 | 最低要求 |
|------|----------|
| `SLAM-文献树.md` | 为本批论文补充 `[[笔记名]]` 与一句贡献/路线归类；按技术路线与 S/A/B/C 分档维护，避免重复链接 |
| `SLAM-挑战洞察树.md` | 将本批论文挂到至少一个 Challenge→Insight 节点下，或新增节点 |
| `SLAM研究知识库.md` | 更新入口说明；可增「近期批量入库」小节，链向双树与代表性 `[[笔记]]`（不必长篇粘贴） |

说明：

- 若用户目录**明显非 SLAM**（例如纯 NLP），可跳过三文件内容合并，但须在 `SLAM研究知识库.md`（或用户指定的总览）**写一句范围说明**，避免误归档。
- 若本批**全部失败**，跳过本节内容合并，并在汇总中说明。

### 4.2 刷新目录索引（MOC）

若 `AUTO_REFRESH_INDEXES=true`，执行：

```bash
python3 ../_shared/generate_concept_mocs.py
python3 ../_shared/generate_paper_mocs.py
```

### 4.3 Git（可选）

1. 只有在 `GIT_COMMIT_ENABLED=true` 时才考虑 git：
   - 先确认 `VAULT_PATH/.git` 存在
   - `git add -A` 后必须真的有 staged changes
   - 满足条件才允许 commit
2. 若 `GIT_PUSH_ENABLED=true` 且存在远端，再执行 push

补充：

- 若本次是分批运行（`limit < total`），**每一批结束**都应执行 **4.1 + 4.2**（至少合并本批增量 + 刷新索引），保证总览与 MOC 持续可见。
- 若全部失败，则跳过 4.1/4.2 并明确提示失败原因。

## Step 5: 输出结果（简短汇总）

完成后只返回精简统计，不展开冗长中间过程：

- 目标目录
- 扫描到的文件总数 / 可处理论文数 / 跳过数
- 成功生成笔记数
- 失败数与失败文件清单
- 概念库是否已更新
- **SLAM 三总览是否已更新**（`SLAM-文献树.md` / `SLAM-挑战洞察树.md` / `SLAM研究知识库.md`；非 SLAM 目录则说明是否已写范围说明）
- 索引是否已刷新

如果启用了分批参数，额外输出：
- 本批处理数量 / 剩余待处理数量
- 是否可继续 `resume`

## 注意事项

- 这是“目录批量入口”，单篇阅读细节与质量规则全部继承 `paper-reader`。
- 不要求用户先手动运行其他内部步骤，默认直接完整执行。
- 若目录非常大，可按用户要求增加限制（如“只处理前 N 篇”或“只处理最近修改的文件”）。
- 如果用户明确要求“使用 Cursor，不使用 Claude CLI”，必须遵守；任何 `claude` 命令都视为违规实现。
